# 全市场策略信号对齐扫描设计

## 目标

为 Yansir 增加一个服务端全局策略扫描器。扫描器按自然时钟每 5 分钟对齐执行，覆盖全市场支持币种，并在对应 K 线收盘后检查 `5m`、`15m`、`30m`、`1h`、`4h` 策略是否命中。命中信号继续使用现有信号保存、用户匹配、收件箱和推送链路。

## 已确认的产品规则

- 扫描时刻对齐自然时钟：`00、05、10、15……55` 分。
- 每个扫描点在边界后保留 5 秒收线缓冲，避免读取尚未完整封口的 Binance K 线。
- 扫描范围是系统支持的全市场币种，不受单个用户套餐扫描上限影响。
- 支持周期为 `5m`、`15m`、`30m`、`1h`、`4h`。
- 各周期只在自己的 K 线收盘点运行，不在每个 5 分钟批次重复计算长周期未收盘 K 线。
- 命中后执行完整生命周期：写入全局信号池、匹配用户自选、写入用户 inbox，并按套餐、告警阈值、冷却和推送配置投递。

## 调度规则

扫描器以 UTC 时间戳判断 K 线边界，因此服务器位于任何时区都得到同一批市场收线结果。

| 当前边界 | 本批周期 |
| --- | --- |
| 每个 5 分钟边界 | `5m` |
| 分钟可被 15 整除 | 加入 `15m` |
| 分钟可被 30 整除 | 加入 `30m` |
| 每个整点 | 加入 `1h` |
| UTC 小时可被 4 整除的整点 | 加入 `4h` |

示例：

- `14:05`：`5m`
- `14:15`：`5m + 15m`
- `14:30`：`5m + 15m + 30m`
- `15:00`：`5m + 15m + 30m + 1h`
- `16:00 UTC`：`5m + 15m + 30m + 1h + 4h`

每次运行结束后重新计算下一个绝对边界并使用一次性定时器等待，避免 `setInterval` 因执行耗时造成逐步漂移。若某批仍在运行，下一个触发点记录为跳过，不并发启动第二批。

## 架构边界

新增的全局扫描器与现有用户可启动/停止的内存定时任务分开：

- 全局扫描器是系统级任务，API 启动后自动开启。
- 用户定时任务继续遵守 API 权益，并保留现有接口行为。
- 全局扫描器复用现有市场币种发现、策略执行、信号持久化、用户 inbox 匹配和告警投递能力，不复制策略规则。
- 调度计算保持为无副作用的独立函数，便于用固定时间测试。

建议在 `StrategyService` 外提取轻量的调度计算模块，并由 `StrategyService` 负责实际批次执行。这样时间计算不依赖网络、数据库或 Nest 生命周期。

## 数据流

1. API 模块启动，计算下一个五分钟边界及 5 秒缓冲后的触发时间。
2. 到时后生成批次标识：`global:<bar-close-iso>`。
3. 根据边界选择本批需要执行的周期。
4. 从现有市场服务获取全市场支持币种。
5. 按周期和币种执行现有策略；单币种失败只记录错误，不中断整批。
6. 策略返回的正式信号通过现有 `SignalsService` 保存。
7. `signal_events.dedupe_key` 的唯一约束保证服务重启后重复计算同一根 K 线时不会产生第二条正式事件。
8. 新事件继续进入现有用户自选匹配、inbox 和告警投递链路；已存在事件不重复投递。
9. 保存内存运行摘要，并计算下一次绝对触发时间。

## 去重与并发

- 进程内使用 `running` 和最近批次键，防止同一实例重入。
- 信号级跨重启去重沿用数据库 `signal_events.dedupe_key` 唯一约束。
- 推送级去重沿用 `alert_deliveries(user_id, signal_event_id, channel)` 唯一索引。
- 本次不引入 Redis、队列或多实例分布式锁。多 API 实例可能重复执行策略计算，但不会重复生成正式信号或用户推送；分布式执行互斥留作生产扩展。

## 状态与可观测性

状态输出至少包含：

- `enabled`、`running`
- `lastSlotAt`、`lastStartedAt`、`lastFinishedAt`
- `nextRunAt`
- `lastTimeframes`
- `scannedSymbols`、`matchedSignals`、`failedSymbols`
- `skippedOverlappingRuns`
- 精简的最近错误列表

状态接口只暴露运行摘要，不返回大量逐币种策略结果。雷达页面可以用这些字段区分“监听中”“本轮无信号”“部分失败”和“扫描暂停”。

## 错误处理

- 币种列表获取失败：本批标记失败，保留错误，继续安排下一批。
- 单个币种或周期失败：累计失败计数，其余任务继续。
- 数据库不可用：沿用现有降级行为，但状态必须明确标记未持久化；不能把未保存的结果伪装成正式命中。
- 扫描超过下一个边界：跳过重叠批次，不并发堆积。
- 服务关闭：清除一次性定时器，不再启动新批次。

## 测试策略

测试按测试驱动方式实现，至少覆盖：

- 任意当前时间都能计算到下一个五分钟边界加 5 秒。
- `:05`、`:15`、`:30`、整点和四小时整点选择正确的周期组合。
- 执行耗时不会让后续触发逐轮漂移。
- 同一实例不会并发执行两个批次。
- 单币种失败不会阻断整批。
- 相同 K 线的重复策略结果不会生成重复 `signal_event` 或重复推送。
- 命中信号会进入全局信号池、用户 inbox 和现有告警规则链路。
- 状态接口正确返回下一次扫描和最近结果。

## 非目标

- 不修改策略判定逻辑或评分阈值。
- 不新增前端页面。
- 不把全局任务绑定到某个用户套餐。
- 不在本次引入 Redis、消息队列或分布式调度平台。
- 不移除现有实时 WebSocket K 线监听或用户定时扫描接口。

## 验收标准

- 服务启动后无需人工点击，扫描器自动按五分钟自然时钟运行。
- 五种周期只在各自收线点扫描全市场。
- 单批故障可观测且不会停止后续批次。
- 相同 K 线不会产生重复正式信号、用户 inbox 或推送。
- 现有手动扫描、用户定时任务、实时监听和雷达信号展示保持兼容。
